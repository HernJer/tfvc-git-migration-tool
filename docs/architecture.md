# Pipeline Architecture

The `tfvc2git` tool is designed around a decoupled, 4-stage pipeline. By breaking the migration into distinct steps, the tool can gracefully recover from interruptions, execute expensive API calls independently of local Git operations, and provide robust audit trails.

## 1. Export (`Export-TfvcChangeset`)

**Goal**: Query the TFVC API, extract the changeset history, and build an intermediate metadata representation.

- **Process**: 
  - Iterates over the `sourceMappings` defined in `config.json`.
  - Queries the Azure DevOps / TFS REST API for the history of the mapped paths.
  - Generates a deduplicated, chronological list of changesets.
  - Outputs the list to `migration-output/changesets.json`.
- **Resiliency**: If a network error occurs, the export can be resumed using `--Resume`. The tool will read `export-checkpoint.json` and pick up where it left off.
- **Note**: No files are downloaded during this step. Only metadata (author, comment, timestamp, and affected file paths) is retrieved.

## 2. Replay (`Invoke-TfvcReplay`)

**Goal**: Translate the TFVC changeset metadata into physical Git commits.

- **Process**:
  - Initializes a new, empty Git repository in `migration-output/git-repo`.
  - Sorts target branches topologically to ensure parent branches (e.g., `main`) are created and populated before dependent child branches (e.g., `develop`).
  - Iterates through `changesets.json` sequentially.
  - For each changeset, downloads the physical file blobs from TFVC for the exact version at that point in time.
  - Stages the files in the local Git repository.
  - Commits the files using the original TFVC author, timestamp, and comment.
- **Resiliency**: Downloads happen in batches. If the replay fails (e.g., due to local disk space issues or API throttling), running it again with `--Resume` will read `replay-checkpoint.json` and skip the changesets that were already successfully committed.

## 3. Verify (`Test-TfvcMigration`)

**Goal**: Cryptographically guarantee that the resulting Git repository perfectly matches the state of the source TFVC repository.

- **Process**:
  - Uses a robust 3-pass verification engine.
  - **Pass 1 (Inventory)**: Compares the list of files in the latest TFVC changeset against the current `HEAD` of the corresponding Git branch. Identifies any orphaned, missing, or unexpected files.
  - **Pass 2 (Hashes)**: Downloads the latest blob of every tracked file directly from TFVC to a temporary directory. Computes the SHA-256 hash. Computes the SHA-256 hash of the corresponding file in the local Git working copy. Compares the hashes to guarantee bit-for-bit accuracy.
  - **Pass 3 (Coverage)**: Verifies that every changeset extracted in Step 1 maps perfectly to exactly one commit hash in the final Git repository.
- **Resiliency**: The downloaded temporary files are immediately deleted after hashing to conserve disk space.

## 4. Report (`New-TfvcMigrationReport`)

**Goal**: Produce a human-readable, non-repudiable HTML audit report.

- **Process**:
  - Reads the raw JSON and CSV data generated by Step 3.
  - Parses `migration-log.txt` to calculate total migration duration.
  - Generates a standalone `audit-report.html` file that can be distributed to stakeholders, compliance teams, or attached to migration tracking tickets.

## Summary

This decoupled architecture allows you to run specific steps independently using subcommands like `tfvc2git export` or `tfvc2git verify`. When you execute the default `tfvc2git run` command, all four steps are orchestrated sequentially.
